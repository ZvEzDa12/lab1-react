# Шпаргалка к защите (React + Vite)

Здесь — подробные объяснения и аналогии из жизни, чтобы было проще уложить в голове, «что к чему».

---

## 1) Что такое компонент

### Простыми словами

Компонент в React — это кусок интерфейса (кнопка, форма, блок с текстом) плюс логика, которая за ним стоит.  
Обычно это функция, которая возвращает JSX — код, похожий на HTML, но внутри JavaScript/TypeScript.

### Аналогия из жизни

Представь деталь конструктора LEGO: у неё есть своё место, свой вид и свои «правила сборки».  
Ты можешь вставить такую деталь в разные модели — сама деталь остаётся понятной и отдельной от остального.

Или виджет на телефоне: погода, календарь — каждый сам по себе, но вместе они составляют экран.

Компонент — это как раз такая «деталь экрана»: отдельный, обозримый кусок.

### В твоём проекте

Пример компонента: `App` в `src/App.tsx`.  
Сейчас почти весь экран живёт внутри одного компонента `App` — это нормально для небольшой лабы.

---

## 2) Можно ли переиспользовать компонент

### Простыми словами

Да. Это одна из главных идей React: написал один раз — используешь много раз.

Переиспользование делается через:

- `props` — разные «настройки» при каждом использовании;
- композицию — вкладываешь компоненты друг в друга, как матрёшки.

### Аналогия из жизни

Один и тот же шаблон бланка в поликлинике: поля одни и те же, но в одном случае пишут «анализ крови», в другом — «УЗИ».  
Шаблон один, данные разные — это как один компонент с разными `props`.

### Пример из твоей темы (как могло бы быть)

Один универсальный компонент `NumberField` можно было бы использовать для возраста, роста и веса, передавая разные `label`, `value`, `onChange`, `error`.  
В коде сейчас поля могут быть прописаны прямо в `App` — но идея переиспользования именно такая.

---

## 3) Что такое props

### Простыми словами

`props` (от *properties* — свойства) — это входные данные, которые родитель передаёт дочернему компоненту.  
Можно думать: аргументы функции, только снаружи их передают так: `<Child name="Анна" />`.

### Аналогия из жизни

Заказ в кафе: ты говоришь официанту «капучино, без сахара».  
Официант не придумывает напиток сам — он передаёт на кухню то, что ты заказал.  
Для компонента `props` — это твой заказ: родитель сказал, что показывать и с какими данными.

Или наклейка на посылке: «кому», «откуда», «хрупкое» — курьер не меняет текст наклейки, он использует то, что написали.

### Пример кода

```tsx
function Greeting({ name }: { name: string }) {
  return <h1>Привет, {name}</h1>
}
```

Тут `name` — это `props`. Родитель передаст: `<Greeting name="Маша" />`.

### Где props «живут» в компонентах

- В функциональном компоненте — в параметрах функции:
  - `function MyComp(props) { ... }`
  - или деструктуризация: `function MyComp({ title }) { ... }`
- В классовом — через `this.props`.

---

## 4) Можно ли менять props внутри компонента

### Коротко

Нет, напрямую нельзя (и так задумано).

### Простыми словами

`props` в React — только для чтения (*read-only*).  
Их меняет родитель (новый рендер с новыми значениями), а не дочерний компонент «внутри себя».

### Аналогия из жизни

Рецепт на кухне: тебе дали бумажку «200 г муки».  
Ты не перечёркиваешь на чужой рецепте «пусть будет 300» — это не твоя бумажка.  
Если нужно другое количество — говоришь тому, кто рецепт выдал (родителю), он выдаёт новый рецепт.

В React: если нужно изменить данные — либо `state` у себя, либо `onChange` / callback наверх, родитель обновит данные и снова передаст новые `props`.

### Правильная схема

- Нужно что-то хранить и менять только здесь → `useState` (state).
- Нужно изменить данные у родителя → вызываешь функцию из props (например, `onChange`), родитель делает `setState`, и ты получаешь новые `props`.

---

## 5) Что можно менять в компоненте

### Можно

- `state` через `setState` из `useState` (в функциональных компонентах);
- вычислять новые значения на каждом рендере (переменные внутри функции);
- обрабатывать события: `onClick`, `onChange`, `onSubmit` — и внутри вызывать `setState` или колбэки;
- менять то, что рисуется, в зависимости от state и props.

### Нельзя

- Переписывать `props` как свои переменные (нарушает правила React и ломает предсказуемость).
- Менять state «в лоб» без setter: нельзя `count = count + 1` для state из `useState`, нужно `setCount(count + 1)` — иначе React может не узнать об изменении.

### Аналогия

State — как счётчик очков на табло: меняешь только кнопкой «+1», которую выдал зал (это `setState`), а не маркером прямо по экрану произвольно.

---

## 6) Что такое состояние приложения (state)

### Простыми словами

`state` — это данные, которые могут меняться и от которых зависит картинка на экране.  
Когда state меняется, React перерисовывает компонент (и при необходимости детей).

### Аналогия из жизни

Доска объявлений в подъезде: сегодня одно объявление, завтра другое — содержимое доски изменилось, все видят новое.  
State — это как текущее содержимое доски: поменял — интерфейс должен показать актуальное.

Или светофор: горит красный / жёлтый / зелёный — одно состояние в каждый момент, от него зависит, что видят водители.

### В твоём `App.tsx` к state относятся

- поля формы: `gender`, `age`, `height`, `weight`, `activity`;
- ошибки: `ageError`, `heightError`, `weightError`;
- результаты расчёта: `basalNorm`, `weightMaintenanceNorm`;
- флаг показа результата: `isResultVisible`.

Всё это — «память» экрана на время работы с формой.

---

## 7) На чем взаимодействует React

### Простыми словами

React держит в памяти виртуальное описание интерфейса (Virtual DOM), сравнивает новое с предыдущим и точечно обновляет настоящий DOM в браузере — не перерисовывая страницу целиком без нужды.

### Аналогия из жизни

Черновик и чистовик: ты сначала правишь черновик (быстро, в голове или на бумаге), смотришь «что именно поменялось», и только потом вносишь точечные правки в документ.  
Не переписываешь весь диплом из-за одной запятой.

Или навигация: перестроил маршрут в голове, потом один раз повернул руль — а не разобрал машину.

Virtual DOM — «черновик интерфейса в памяти»; реальный DOM — то, что видит пользователь в браузере.

### Итог

Интерфейс обновляется эффективно: меняется в основном то, что действительно зависит от новых данных.

---

## 8) Функциональный и классовый компонент: отличия

### Функциональный компонент

- Это обычная функция.
- Состояние и «побочные эффекты» — через хуки: `useState`, `useEffect`, `useMemo`, `useCallback`.
- Сейчас это обычный стандарт в новых проектах.

Аналогия: один современный прибор с кнопками «режим 1 / режим 2» — всё в одной коробке, без отдельной инструкции на 50 страниц.

### Классовый компонент

- Это класс `extends React.Component`.
- Состояние: `this.state`, обновление: `this.setState`.
- Жизненный цикл: `componentDidMount`, `componentDidUpdate`, `componentWillUnmount` и т.д.

Аналогия: старый аппарат с рычагами и отдельными фазами «включение — работа — выключение» — то же самое по смыслу, но другой «язык описания».

### Ключевые различия (таблица в голове)

| | Функциональный | Классовый |
|---|----------------|-----------|
| Синтаксис | `function App() { ... }` | `class App extends Component` |
| Состояние | `useState` | `this.state` / `setState` |
| Эффекты | `useEffect` | lifecycle-методы |
| Обычно в новых проектах | Чаще | Реже |

---

## 9) Зачем нужны функциональные компоненты

### Простыми словами

Их выбирают, потому что код короче, привычнее в экосистеме React сегодня и удобнее выносить повторяющуюся логику в свои хуки (кастомные функции с `use`).

### Аналогия

Вместо толстой тетради с уставом — один блокнот и закладки: нужный кусок логики вытаскиваешь отдельно (`useSomething`), не копируя полкласса.

## 10) Как работает твоя лабораторная (подробно)

### По шагам «как в жизни»

1. Человек заполняет анкету — вводит возраст, рост, вес, выбирает пол и активность.  
   В коде: пользователь меняет поля формы.

2. Каждое изменение попадает в state — как будто секретарь сразу заносит правки в рабочую таблицу.  
   В коде: обработчики `onChange` вызывают `setState`.

3. Проверка полей — «возраст не отрицательный», «рост разумный» и т.д.  
   В коде: валидация выставляет `ageError`, `heightError`, `weightError`.

4. Кнопка «Рассчитать» доступна только когда форма валидна — как касса: не пробьёт чек, пока не заполнены обязательные поля.

5. Нажатие «Рассчитать» — запускаются формулы (базовый обмен, норма с учётом активности).  
   В коде: `onSubmit`, функции вроде `calculateBasalCalories`, результаты кладутся в state.

6. Показ результата — когда `isResultVisible === true`, блок с цифрами появляется (или становится видимым).  
   Как шторка на сцене: сначала готовили, потом показали.

7. «Очистить» — сброс к начальным значениям, как новый бланк.

### Краткая цепочка для защиты

Ввод → state → валидация → submit → расчёт → сохранение результата в state → условный показ результата → очистка по кнопке.

## 11) Готовые короткие ответы преподавателю

Что такое компонент?Независимый блок UI и логики, обычно функция, возвращающая JSX.  
*Аналогия:* деталь конструктора или виджет на экране.

Можно ли переиспользовать? Как?Да — через props и композицию: один компонент, разные входные данные и вложенность.

Что такое props?Входные данные от родителя, как параметры или заказ на кухню — только читать, сам список заказа дочерний не переписывает.

Можно ли менять props внутри компонента?Нет. Меняют state или просят родителя через callback.

Что такое state?Изменяемые данные, от которых зависит отображение; при изменении — перерисовка.  
*Аналогия:* текущее состояние светофора или текста на доске объявлений.

На чем React взаимодействует с браузером?Через Virtual DOM и точечное обновление реального DOM.

Чем классовый отличается от функционального?Класс: `this.state`, lifecycle. Функция: хуки (`useState`, `useEffect`). Сейчас чаще функциональные.

## 12) Базовые вещи React — подробнее и с аналогиями

### JSX

Что это: синтаксис, похожий на HTML, но внутри JS/TS. Браузер его не понимает напрямую — его переводят в вызовы React.

Аналогия: сценарий спектакля в понятной форме — актёрам удобно читать, а в зале уже «настоящее» представление (DOM).

---

### Однонаправленный поток данных

Суть: данные текут сверху вниз — от родителя к ребёнку через props.  
Ребёнок не ломает переданное; если нужно изменить данные «наверху», он зовёт функцию (callback).

Аналогия: приказы сверху вниз в организации; отчёт снизу вверх — не переписываешь приказ у себя в ящике, а отправляешь отчёт начальнику, он выпускает новый документ.

---

### Ререндер

Компонент перерисовывается, когда меняются его state, его props или state родителя (часто перерисовываются и дети).

Аналогия: перерисовка — как обновить фото на витрине, когда в магазине привезли новый товар: витрина та же по месту, содержимое другое.

---

### События

`onClick`, `onChange`, `onSubmit`, `onBlur` — это реакции на действия пользователя.

Аналогия: звонок в дверь — ты открываешь (обработчик срабатывает) и делаешь что-то дальше (обновляешь state).

---

### Условный рендеринг

Показываешь кусок UI только если условие истинно.

Примеры из проекта:

- `{ageError && <span>...</span>}` — ошибка есть только когда `ageError` не пустой;
- видимость блока результата зависит от `isResultVisible`.

Аналогия: наклейка «скидка» на товар — приклеена только если акция действует.

---

### Списки и `key`

Когда рисуешь массив элементов (`map`), у каждого должен быть стабильный `key` — чтобы React понимал, какой элемент какой, при обновлениях.

```tsx
items.map((item) => <li key={item.id}>{item.name}</li>)
```

Аналогия: номера мест в театре — не «третий слева в каком-то ряду без номера», а конкретное место 12, чтобы при пересадке не перепутать людей.

---

### Хуки (кратко)

| Хук | Задача | Аналогия |
|-----|--------|----------|
| `useState` | хранить меняющиеся данные | ячейка памяти с кнопкой «обновить» |
| `useEffect` | код после отрисовки, побочные дела | «когда сцена готова — включи свет / запроси данные» |
| `useMemo` | не пересчитывать тяжёлое без нужды | записать результат в тетрадь, а не считать заново каждый раз |
| `useCallback` | не создавать новую функцию каждый рендер без нужды | держать ту же ссылку на функцию для детей |

В твоём коде есть `useState` и `useCallback`.

---

### `useEffect` простыми словами

Запускает функцию после того, как React отрисовал экран (в определённые моменты, которые ты задаёшь зависимостями).

Зачем: запросы к API, подписки, таймеры, работа с DOM вне React.

Аналогия: «когда почта доставлена — открой конверт» — не в момент, пока ты ещё пишешь адрес на конверте.

---

### Controlled components (управляемые поля)

Значение поля берётся из state, onChange обновляет state. Источник правды — React, а не «свободный» DOM.

Аналогия: электронная таблица в облаке — все видят одно и то же число; поменял ячейку — обновилось у всех. Не два разных «черновика» у браузера и у React.

В твоей форме так связаны `age`, `height`, `weight` и остальные поля.

---

### Почему React удобен

- Дробление на компоненты — проще думать и искать ошибки;
- Переиспользование;
- Предсказуемость: state → UI;
- Масштабирование больших интерфейсов.

Аналогия: город из кварталов проще администрировать, чем один сплошной «ком».

---

### Частые ошибки новичков

- Пытаться менять props «как переменные»;
- Менять state без `setState` / setter из `useState`;
- Забывать `key` в списках;
- Делать огромный один файл вместо разбиения на компоненты (для лабы простительно, в большом проекте — больно).

---

## 13) Мини-схема «как всё связано» (для защиты)

```
Родитель
  │  передаёт props (данные + функции-callback)
  ▼
Дочерний компонент
  │  читает props, держит свой state (если нужен)
  │  пользователь кликает / вводит → события
  ▼
setState / callback родителю
  │
  ▼
React пересчитывает Virtual DOM → обновляет реальный DOM точечно
```

Если запомнишь одну фразу: props сверху, события снизу вверх (через callback), state — твоя память экрана — уже многое для ответов на вопросы будет на месте.
